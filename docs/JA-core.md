상태: active / improving / resolved

### 예상 약점 (프로필 기반 추정, 학습 후 검증)
- 조사 사용 (は/が/を/に/で 구분)
- 동사 활용형 (ます형, て형, ない형 전환)
- 장음/촉음 구분 (おばさん vs おばあさん, きて vs きって)
- 한자 음독/훈독 혼동
- 어순 (한국어와 비슷하지만 조사 위치가 다른 경우)
- 경어/반말 레벨 혼용

<!-- §8A-REVIEW -->
## 8-A. 복습 큐 (활성 문장만)

`japanese-data.json` → `reviewQueue` 배열에서 관리한다.

### 항목 스키마 (참조용)

```json
{
  "id": "D1-1",
  "sentence": "すごい！",
  "reading": "すごい",
  "keyExpression": "すごい",
  "category": "A",
  "learnedDate": "2026-03-15",
  "currentInterval": 1,
  "nextReview": "2026-03-16",
  "consecutivePasses": 0,
  "lastResult": null
}
```

### 주기 규칙
- 학습 당일 = Day 0
- 1일 후 → 3일 후 → 7일 후 → 21일 후 → 60일 후
- O(통과): 다음 주기로 이동
- △(부분 통과): 통과 처리하되 현재 주기와 다음 주기의 중간 날짜에 추가 복습 1회 삽입
- X(리셋): Day 0으로 리셋, consecutivePasses 0으로 초기화
- 60일 통과: reviewQueue에서 제거, stats.reviewCompleted60dSentences +1

### 용량 관리
- reviewQueue 길이가 80을 넘으면 신규 학습을 3개로 제한한다.
- 100을 넘으면 졸업 후보를 학습자에게 제시한다.

<!-- §8B-KANJI -->
## 8-B. 한자 학습 큐

`japanese-data.json` → `kanjiQueue` 배열에서 관리한다.

### 항목 스키마 (참조용)

```json
{
  "id": "K1-1",
  "kanji": "学",
  "onyomi": "がく",
  "kunyomi": "まな(ぶ)",
  "koreanReading": "학",
  "mnemonic": "がく = 학 — 거의 같은 소리",
  "exampleWord": "学校(がっこう)",
  "learnedDate": "2026-03-15",
  "currentInterval": 1,
  "nextReview": "2026-03-16",
  "consecutivePasses": 0,
  "lastResult": null
}
```

### 주기 규칙
§8-A와 동일 (1일 → 3일 → 7일 → 21일 → 60일).

### 등록 기준
Step 2에서 새 문장에 포함된 한자 중 학습자가 처음 접하는 것만 등록한다. 이미 kanjiQueue에 있는 한자는 중복 등록하지 않는다.

### 용량 관리
- kanjiQueue 길이가 60을 넘으면 신규 한자 도입을 세션당 2개로 제한한다.
- 80을 넘으면 졸업 후보를 학습자에게 제시한다.

<!-- §10-CATEGORY-TABLE -->
| # | 카테고리 | 설명 | 상태 | 시작일 | 완료일 |
|---|---|---|---|---|---|

| C | 의지/감정 표현 | 俺がやる, 絶対に負けない, 守りたい, 許さない 등. 배틀 애니 핵심 대사 패턴 | 대기 | — | — |

| F | 동사 활용 | ます→て→ない→た→ば/たら. 활용형을 단계적으로 | 대기 | — | — |
| G | 여행 실전 | 식당, 편의점, 역, 호텔, 돌발 상황 | 대기 | — | — |
| H | 애니 구어체 심화 | 축약(～てる, ～ちゃう), 종조사(よ/ね/ぞ/な), 캐릭터별 말투 차이 | 대기 | — | — |

**상태 데이터:** `japanese-data.json` → `categories` 배열. 위 테이블은 참조용이며, 실제 상태는 JSON에서 관리한다.

<!-- §10-CATEGORY-RULES -->
### 진행 규칙
- 카테고리 A, B는 병행 진행한다 (반응 표현 + 기본 문장 구조는 모든 단계의 기반).
- 한 카테고리에서 핵심 표현 20개 이상이 7일 주기(3회차 복습)를 통과하면 AI가 완료 여부를 판단한다. 20개는 최소 기준이며, 카테고리 범위가 넓은 경우(F, H 등) AI 판단에 따라 완료 기준을 늘릴 수 있다. 기준 변경 시 학습자에게 고지한다.
- 완료된 카테고리의 문장도 60일 복습 주기까지 계속 복습 대상에 포함한다.

## 14. AI 행동 규칙

<!-- §14-READ-ORDER -->
### 문서 읽기 우선순위
1. §8-A(복습 큐) + §8-B(한자 큐) — 오늘 복습 대상 파악
2. §6(약점) — 신규 학습에 반영할 약점
3. §10(카테고리) — 현재 진행 중인 카테고리
4. JA-method.md §1(프로필) — 수준 확인 (첫 세션/점검 시)

<!-- §14-GENERATION-RULES -->
### 학습 생성 규칙
- §8-A 복습 큐 전체 행 수가 80을 넘으면 신규 학습을 3개로 제한한다 (아래 규칙보다 우선).
- 복습 대상(문장+한자 합산)이 15개 이상이면 신규 학습을 3개로 줄인다.
- 복습 대상이 5개 미만이면 신규 학습을 8개까지 늘린다.
- 전 세션 약점에 기록된 항목이 있으면 관련 문장을 신규에 1~2개 포함한다.
- 학습자가 "오늘 가볍게"라고 하면 복습만 진행한다.
- 학습자가 "오늘 빡세게"라고 하면 신규를 12개까지 늘린다.
- 학습자가 "넘어가자", "다음"이라고 하면 즉시 다음 단계로 이동한다.
- 한 세션에 새 한자는 최대 5개까지만 도입한다. 한자 과부하를 방지한다.
- 장면 기반 학습 시 대화는 세션당 2개로 제한한다. 신규 표현은 세션당 1~2개.
- 복습 대상이 10개 이상이면 대화를 1개로 줄이고 신규 표현을 1개로 제한한다.
- 대화 안에 학습자가 아직 배우지 않은 단어가 포함되면 안 된다. 모든 단어가 "이미 배운 것" 또는 "오늘의 +1"이어야 한다. 불가피하게 미학습 단어가 필요한 경우, 대화 전에 먼저 그 단어를 소개한다.
- Day 1~10의 대화에는 반드시 이전 세션에서 배운 표현이 자연스럽게 재등장해야 한다.
- 하나의 장면(예: "이타도리의 첫날")은 3~4세션에 걸쳐 이어진다. 장면 전환은 AI가 판단하되 학습자에게 고지한다.

<!-- §14-DIFFICULTY -->
### 난이도 조절 규칙
- 복습 성공률이 최근 3세션 연속 90% 이상이면 다음 카테고리를 병행 시작한다.
- 복습 성공률이 최근 3세션 연속 50% 미만이면 신규 학습을 중단하고 복습에 집중한다.
- 작문에서 같은 실수가 3회 반복되면 해당 문법/패턴을 별도 미니 레슨으로 진행한다.
- "연속"은 날짜가 아니라 세션 기준이다. 이틀 쉬고 3회 학습해도 3세션 연속으로 카운트한다.

<!-- §14-KANJI-INTRO -->
### 한자 도입 규칙
- 카테고리 A(기본 반응 표현)에서는 한자를 적극 도입하지 않는다. 히라가나 중심으로 학습한다.
- 카테고리 B부터 문장에 포함된 기초 한자를 자연스럽게 도입한다.
- 새 한자를 소개할 때 반드시 한국어 한자음을 함께 알려주고 JA-ref.md §13에 기록한다.
- JLPT N5 수준 한자(약 80자)를 우선 커버하되, 애니 빈출 한자가 N5 밖이라도 필요하면 도입한다.

### 학습 로그 용량 관리
- 학습 로그가 30일분을 넘으면 가장 오래된 것부터 요약으로 압축한다.
- 보존 대상: 현재 복습 주기 안에 있는 문장은 절대 삭제하지 않는다.
- 압축 형식: `Day 1~10 요약: 문장 N개 학습, 한자 N개 학습, N개 복습 완료, 주요 약점: [목록]`

<!-- §14-METHODOLOGY -->
### 방법론 점검
- 10일마다 AI가 현재 학습 방법의 효과를 자체 점검한다.
- 복습 성공률, 작문 정확도, 약점 해결 속도, 한자 인식률을 기준으로 판단한다.
- 비효율적인 부분이 있으면 학습자에게 방법 변경을 제안한다.
- 변경은 학습자가 승인한 뒤에만 적용한다.

<!-- §14-ABORT -->
### 세션 중단 처리
- 학습자가 "오늘 여기까지" 또는 중단 의사를 밝히면, 완료된 Step까지만 학습 로그에 기록한다.
- 복습만 진행하고 신규를 안 한 날도 Day 카운트에 포함한다.
- 작업지시서는 실제 완료된 내용만 반영한다.

<!-- §14-CHECKLIST -->
### AI 자주 하는 실수 체크리스트
(학습 진행 중 발견되면 여기에 누적)
- [x] 작업지시서 출력을 먼저 물어봄 — 학습자가 "작업지시서"라고 말할 때만 출력할 것
- [x] 세션 마무리를 먼저 제안함 — 학습자가 종료 의사를 밝힐 때까지 다음 Step으로 진행할 것
- [x] i+1 원칙 위반 — 대화에 미학습 단어를 다수 포함함. 모든 단어가 "이미 배운 것" 또는 "오늘의 +1"이어야 한다
- [x] 한자 불필요 표기 — 한자를 외울 필요 없는 단어를 한자로 표기함. 히라가나로 가르칠 것
- [x] 과다 분량 — 신규 표현 5개 + 대화 5개는 초보에게 과다. 대화 2개, 신규 1~2개로 제한
- [x] ChatGPT Voice에 발음 코칭 역할 부여 — ChatGPT는 발음을 분석하지 못한다. "통하는지 확인" + "딕테이션 출제"만 유효
- [x] 초기 단계에서 ChatGPT Voice 사용 — 단어 수준에서 대화 연습은 무의미. 문장 조립 가능 시점(카테고리 B 중반)까지 중단

## 15. 문서 업데이트 작업지시서 (VS Agent / Haiku용)

### 역할 분담

| 역할 | 담당 | 원칙 |
|---|---|---|
| 학습 진행 + 판단 | Opus 4.6 | 복습 판정, 신규 생성, 약점 분석, 난이도 조절 — **모든 지적 판단** |
| 작업지시서 생성 | Opus 4.6 | 세션 종료 시 판단 결과를 확정값으로 출력 |
| 파일 편집 | Haiku (VS Agent) | 작업지시서를 읽고 JSON을 편집 — **판단하지 않음** |

### 트리거

학습자가 대화 중 **"작업지시서"**라고 말하면, Opus는 즉시 해당 세션의 변경 사항을 반영한 작업지시서를 출력한다.

### 대상 파일

`japanese-data.json` (Google Drive, GAS 경유)

JA-core.md / JA-method.md는 방법론 변경 시에만 수정한다. 매 세션 작업지시서에서는 JSON만 다룬다.

→ 작업지시서 상세 문법, Haiku 실행 규칙, 출력 예시는 JA-ref.md §15-A를 참조한다.

## 16. 이 문서의 관리

이 문서의 목적은 AI가 매 세션 읽고 행동 규칙과 데이터 스키마를 파악하는 것이다. 학습 상태 데이터는 `japanese-data.json`에서 관리한다.

**핵심 원칙:** JA-core.md가 행동 규칙을, JA-method.md가 방법론을, japanese-data.json이 학습 상태를 각각 담당한다.

**JA-core.md 갱신 시점:** §14 행동 규칙이나 데이터 스키마가 변경될 때.
**JA-method.md 갱신 시점:** 방법론이 변경될 때. 매 세션 갱신 불필요.
**japanese-data.json 갱신 시점:** 매 학습 세션 종료 시. §15의 작업지시서 형식을 따른다.

**영어 학습과의 관계:** EN-core.md / EN-method.md / english-data.json과 독립적으로 운영한다. 양쪽 학습을 같은 날 해도 각 JSON은 별도로 업데이트한다. 다만 §14의 방법론 점검 시 양쪽의 학습 부담이 과하지 않은지 AI가 총량을 확인한다.
